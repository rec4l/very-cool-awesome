# 2v2 Mode — Implementation Plan

Adding a 2-versus-2 team mode to the game. This is the largest structural change
the project has taken on: the codebase is hardcoded around a `player1`/`player2`
pair at every layer, and 2v2 forces us to generalize to N players + teams.

This doc is the plan and the record of design decisions. Nothing here is built yet.

---

## 1. Locked design decisions

These were decided up front because each one reshapes the architecture:

| # | Decision | Choice | Consequence |
|---|----------|--------|-------------|
| 1 | **Player data model** | Generalize to **N players (arrays)**, keyed by id + `team` field | Rewrites the `{1,2}` / `p1`,`p2` wire format and every player reference. Future-proofs 3v3, FFA, spectator. |
| 2 | **Team identity** | **Team-tinted outline ring** — keep each player's color + face, add a blue/red ring | Reuses the existing `EntityRenderer` `ringColor` param. No loss of customization. |
| 3 | **Teammate physics** | **Full friendly fire** (teammates collide + wrecking ball hits them), but **toggleable** | Add a `friendlyFire: boolean` to `GameMode` (default `true`). Collision masks branch on the flag. |
| 4 | **Team assignment** | **Auto by join order + swap button** (P1+P2 = Team A, P3+P4 = Team B) | Simplest lobby logic, fewest empty-slot edge cases. |
| 5 | **Maps** | **One map, made larger** (no per-mode map gating) | Bump the single map's dimensions first (see §8 prerequisite), add 4 spawns. |
| 6 | **Disconnects** | **Play on, team down a player**; reconnect is best-effort / later phase | Match continues 2v1. Seamless rejoin needs a persistent client token (see §7). |
| 7 | **Start requirement** | **Allow uneven (min 2)**, structured so **AI bots can fill** empty slots if the host wants | Ready-gate allows < 4. Bot players implement the same input interface (see §7). |
| 8 | **Win score** | **Configurable per-mode; 2v2 = first to 7** | Already lives in `GameMode.winScore`. Just register the new mode's value. |

## 2. Open decisions (revisit during build / playtest)

Smaller calls that don't block the structure but should be made deliberately:

- **Mode selection UX** — proposed: a mode toggle (1v1 / 2v2) sitting next to the
  existing map picker in the lobby, host-only, same pattern as `select_map`.
- **Pickup density** — with 4 players competing for the same boost orbs, the map
  likely needs more orbs. Proposed: scale `pickupPositions` count with player count,
  or just author more on the enlarged map.
- **Spawn layout** — where the 4 spawns sit (e.g. each team staggered fore/back on
  its side, kickoff-style). Needs tuning once the bigger map exists.
- **Team colors for the ring** — exact blue/red h(or pull from a palette). The ring
  must stay distinct from the 12 player colors so it reads as "team," not "another player."
- **Kickoff symmetry** — ball starts center; confirm spawns are mirror-symmetric so
  neither team has a positional edge.
- **Goal ownership / own-goals** — a goal credits the *defending* team's net, not a
  player. Confirm own-goals count for the other team (they will, naturally, since
  scoring is by net side).

## 3. Current architecture (what we're changing)

Where the `player1`/`player2` assumption is baked in:

**`shared/types/index.ts`**
- `GameState` has `p1`, `p2`, `score: {1,2}`, `powerUps: {1,2}`, `wreckingBalls: {1,2}`.
- `GuestPlayer.playerNumber: 1 | 2`; `LobbyPlayer.playerNumber: 1 | 2`.
- `PlayerStyle` payload + `names` event hardcode `{1, 2}`.
- `MapDefinition.playerStarts: [Vec2, Vec2]` — fixed 2-tuple.
- `MapGoal.scoringPlayer: 1 | 2`.

**`server/src/physics.ts`**
- `createPhysics()` returns `{ player1, player2, ball, ... }` — two named bodies.
- Collision categories: only `CAT_PLAYER1` (0x0001) and `CAT_PLAYER2` (0x0010).
- Wall/ball masks list both player categories explicitly.
- Wrecking ball "don't hit owner" uses `opponentCat = playerNumber === 1 ? CAT_PLAYER2 : CAT_PLAYER1`.
- `resetPositions()` sets exactly two players from `playerStarts[0]`/`[1]`.

**`server/src/rooms.ts`**
- `Room.inputs/score/powerUps/ready` are all `{1, 2}` objects.
- `joinRoom` hardcodes `playerNumber: 2` for the second player.
- `physics` returns the two-body bundle.

**`server/src/game.ts`**
- `buildGameState()` reads `player1`/`player2` into `p1`/`p2`.
- Loop bodies everywhere do `pNum === 1 ? r.physics.player1 : r.physics.player2`.
- `startGame` emits `names: {1, 2}`.
- Goal/win logic keys `r.score[scorer]` by player number.

**`server/src/index.ts`**
- `player_ready` gate: `if (room.ready[1] && room.ready[2])`.
- `disconnect`/`leave_room` → emits `opponent_disconnected` (ends game).

**Client**
- `Game.ts`: `p1Renderer`/`p2Renderer`, `myPlayer: 1|2`, `playerStyles: {1,2}`.
- `hud.ts`: two score texts, single local boost bar / pip row.
- `index.html`: two hardcoded lobby cards (`lobby-ball-1`, `lobby-ball-2`).
- `EntityRenderer`: **already** accepts `ringColor` — repurpose for team ring.

## 4. Target data model

Replace the `{1, 2}` keying with arrays. Core shape:

```ts
type Team = 'A' | 'B';

type Player = {
  id: string;            // socket id (or bot id)
  slot: number;          // 0..3 stable index within the room
  name: string;
  color: number;
  faceId: string;
  team: Team;
  isBot: boolean;        // §7 — bots share the input interface
};

// GameState becomes array-based:
type GameState = {
  players: Array<{
    slot: number;
    pos: Vec2;
    team: Team;
    powerUps: { boostBar; teleportCooldown; teleportCharges; teleported };
    wreckingBall: { active; position: Vec2; velocity: Vec2 };
  }>;
  ball: Vec2;
  score: { A: number; B: number };   // team score, not player score
  matchSeconds: number;
  pickups: Array<{ x; y; active }>;
};
```

Physics returns `players: Matter.Body[]` (length = player count) instead of two
named fields; the ball stays singular.

> **Note on the data-model choice:** because we picked "generalize to N," the same
> arrays trivially support 1v1 (2 players, teams A/B of 1), 2v2 (4 players), and
> later 3v3/FFA. 1v1 becomes "the N=2 case," not a separate code path — which means
> **we refactor 1v1 onto the new model too**, and there's no `{1,2}` left anywhere.

## 5. Phased implementation plan

Ordered so each phase is independently testable and 1v1 never breaks.

### Phase 0 — Prerequisite: enlarge the map (§8)
Bump the single map's `width`/`height` and re-tune wall/goal/spawn geometry for
1v1 first. Ship and confirm 1v1 still feels right at the larger size. This
de-risks 2v2 by separating "bigger arena" from "more players."

### Phase 1 — Generalize the model behind 1v1 (no visible change)
The big, invisible refactor. Goal: 1v1 plays identically, but internally everything
is array/team-based.

- **`shared/types`**: introduce `Team`, array-based `GameState`, `Player.team`/`slot`.
  Change `MapDefinition.playerStarts` → `Vec2[]`. Change `MapGoal.scoringPlayer` →
  `scoringTeam: Team`. Update `PlayerStyle`/`names` event to an array.
- **`shared/modes.ts`**: add `teamSize`, `friendlyFire` to `GameMode`. Keep
  `CLASSIC_1V1` (teamSize 1, maxPlayers 2). Register `TEAM_2V2` (teamSize 2,
  maxPlayers 4, winScore 7, friendlyFire true) but don't expose it yet.
- **`server/src/physics.ts`**: `createPhysics(map, playerCount)` builds an array of
  player bodies. Replace the two fixed collision categories with a per-player
  category scheme (see §6). `resetPositions` loops `playerStarts`.
- **`server/src/rooms.ts`**: `inputs/powerUps` become arrays indexed by slot;
  `score` becomes `{A, B}`; `ready` becomes per-slot. `joinRoom` assigns next slot.
- **`server/src/game.ts`**: every `pNum === 1 ? player1 : player2` becomes a loop
  over `r.physics.players[slot]`. `buildGameState` maps the array. Win check reads
  `r.score[team]`.
- **Client**: `Game.ts` holds a `renderers: EntityRenderer[]`; `myPlayer` becomes
  `mySlot`. `hud.ts` reads team scores. Interpolation/snapshot loop over the array.
- ✅ **Acceptance**: 1v1 is byte-for-byte equivalent in feel; no `{1,2}` references remain.

### Phase 2 — Four players on the field
- Lobby accepts up to 4 (`maxPlayers` from mode). Auto-assign teams by join order.
- Spawn 4 bodies; render 4 players with **team rings** (decision #2).
- Team scoring end-to-end; HUD shows Team A vs Team B.
- ✅ **Acceptance**: 4 humans can play a full 2v2 to 7, with correct team scores.

### Phase 3 — Team UX & identity
- Lobby: two team columns (A/B) replacing the two hardcoded cards; **swap button**.
- Mode toggle (1v1/2v2) next to the map picker, host-only.
- Team ring colors finalized; ensure they're distinct from the 12 player colors.
- HUD polish: team-colored score, optionally teammate boost indicator.
- ✅ **Acceptance**: host can pick mode, players can swap teams, teams read clearly.

### Phase 4 — Friendly fire + collision rules
- Wire `friendlyFire` flag through collision masks (§6). Default on.
- ✅ **Acceptance**: with FF on, teammates collide and WB hits them; flipping the
  flag off makes teammates pass-through to WB but still bump (or fully configurable).

### Phase 5 — Disconnects: play on, down a player
- Replace blanket `opponent_disconnected` with: remove the player's body, keep the
  loop running, broadcast updated roster. End the match only if a whole team empties.
- ✅ **Acceptance**: one player quitting a 2v2 leaves a live 2v1; match still resolves.

### Phase 6 — (Optional) Reconnect + bots
- **Reconnect**: persistent client token (§7) maps a returning socket to its slot.
- **Bots**: fill empty slots with AI that produces `PlayerInput` each tick (§7).
- ✅ **Acceptance**: host can add a bot to fill a slot; a dropped player can rejoin.

## 6. Collision categories — the gnarly part

Matter.js collision filters are a 32-bit category + mask. Today only 2 categories
are used. With 4 players we need each player individually addressable (so the
wrecking ball can exclude exactly its owner), **and** team grouping for friendly fire.

**Proposed scheme** — one category bit per player slot:

```
CAT_BALL    = 0x0001   // soccer ball
CAT_WALL    = 0x0002
CAT_PLAYER0 = 0x0010
CAT_PLAYER1 = 0x0020
CAT_PLAYER2 = 0x0040
CAT_PLAYER3 = 0x0080
```

- **Player body mask**: `CAT_BALL | CAT_WALL | (all other player cats)` — players
  collide with everyone. (Teammate bump = on, matching decision #3.)
- **Wrecking ball mask** (owned by slot *k*):
  - Friendly fire **on**: `CAT_WALL | CAT_BALL | (all player cats except k)` → hits
    opponents *and* teammates, never its own owner.
  - Friendly fire **off**: `CAT_WALL | CAT_BALL | (opponent-team cats only)` → hits
    opponents only. Compute opponent set from each player's `team`.
- The `friendlyFire` flag is read at **deploy time** (`deployWreckingBall`) to pick
  the mask, so toggling the mode config is all it takes.

> **Why per-slot bits, not per-team bits:** the WB must exclude its *owner specifically*
> while still (optionally) hitting the owner's teammate. Per-team bits can't express
> "everyone on my team except me." Four slots fit comfortably in the category space.

## 7. The hard parts you flagged

### Reconnect (you asked: "rejoining might be hard, wouldn't it?")
Yes — this is the genuinely hard bit, which is why it's Phase 6 / optional. The
problem: a reconnecting browser gets a **new socket id**, so the server can't tell
it's the same human returning to a vacated slot.

**Solution sketch:**
1. On first join, server generates a `playerToken` (uuid) and sends it to the client,
   which stores it in `localStorage`.
2. On reconnect, the client sends `{ token }` with its join/handshake.
3. Server keeps the vacated slot reserved (with a `reconnectTimer`, like the existing
   10s room timer) and matches the token to re-seat the player into the same slot/team.
4. If the timer expires first, the slot frees up (or a bot can fill it).

This is self-contained and doesn't block Phases 1–5; "play on, down a player" works
without it. We can ship 2v2 fully playable, then add rejoin as a polish pass.

### AI bots (fill empty slots)
The model makes this clean *if* we design for it now: a `Player` has `isBot: boolean`,
and **bots produce the same `PlayerInput` shape** that humans send over the socket.
- For human slots, `r.inputs[slot]` is set by the `input` socket event.
- For bot slots, a `computeBotInput(state, slot)` function fills `r.inputs[slot]`
  each tick before the physics step — same downstream code, no special-casing.
- A trivial first bot: move toward the ball, boost when far, occasional WB. Can grow
  later. Host adds/removes bots from empty lobby slots.
- **Structural requirement to bake in now:** never assume `r.inputs[slot]` comes from
  a socket; always read it as plain data. (Phase 1 already does this.)

## 8. Prerequisite (aside): enlarge the single map

You want the one map bigger regardless. Doing this **before** 2v2 is the right order
because it isolates the "bigger arena" tuning from the "more players" change.

Touch points:
- `MapDefinition.width`/`height` on the map → larger values.
- Re-derive wall geometry, goal mouth positions, corner bevels, center markings,
  and `pickupPositions` for the new size.
- Client arena canvas already scales off `map.width`/`map.height`, so rendering
  follows automatically; double-check HUD panel layout (anchored off `W`/`H`).
- Re-tune feel constants if needed (player speed / friction may feel different with
  more travel distance).
- Ship + confirm 1v1 at the new size before starting Phase 1.

## 9. Effort & risk

| Phase | Effort | Risk |
|-------|--------|------|
| 0 — Enlarge map | Small | Low (isolated, visible) |
| 1 — Generalize model | **Large** | **High** — touches every file; 1v1 regression risk |
| 2 — Four on field | Medium | Medium — physics/collision correctness |
| 3 — Team UX | Medium | Low — mostly DOM/HUD |
| 4 — Friendly fire | Small | Low — flag + masks |
| 5 — Play-on disconnect | Medium | Medium — room lifecycle edge cases |
| 6 — Reconnect + bots | Large | Medium — token plumbing, bot AI quality |

**Phase 1 is the whole ballgame.** It's a wide, invisible refactor with real
regression risk to the one mode that currently works. Recommend doing it on a branch,
with the existing 1v1 as the acceptance test (it must feel identical), before any
4-player code lands. Everything after Phase 1 is incremental and low-drama.
